# Анализ границ ответственности и транзакционных операций

## Таблица анализа операций

| Операция | Синхронная/ Асинхронная | Откат при ошибке | Retry-стратегия | Идемпотентность | Обоснование |
|:--------|:----------------------:|:----------------:|:---------------:|:---------------:|:-----------|
| **Загрузка файла и сохранение метаданных заметки в БД** | Синхронная | Да (ROLLBACK транзакции БД) | Нет (контролируется БД) | Да (проверка по уникальному ID запроса или хешу файла) | Это ядро операции. Без сохранения заметки дальнейшие действия бессмысленны. Атомарность критична. |
| **Сохранение файла в локальное/облачное хранилище** | Синхронная | Да (удаление загруженного файла) | 3 попытки с эксп. задержкой (1с, 2с, 4с) | Да (хеш файла + имя) | Файл — основной контент. Если его не сохранить, заметка бесполезна. Retry нужен при временных сбоях диска/сети. |
| **Создание записи в сервисе версионирования** | Асинхронная | Нет (помещаем в очередь для повторной попытки) | 5 попыток с эксп. задержкой, затем в очередь мертвых писем (DLQ) | Да (ID заметки + timestamp версии) | Версионирование важно, но не критично для факта создания заметки. Может быть выполнено позже. |
| **Рассылка команд синхронизации на устройства** | Асинхронная | Нет (best-effort через WebSocket/очередь) | Встроено в брокер сообщений (RabbitMQ/Kafka) | Да (уникальный ID события синхронизации) | Если устройство офлайн, оно получит обновление при подключении. Немедленная доставка не гарантируется. |

## Вывод по транзакционным границам

Основная транзакция заканчивается после успешного сохранения файла и метаданных заметки. Все остальные операции (версионирование, синхронизация) вынесены за её пределы и обрабатываются асинхронно, что обеспечивает:
- **Высокую отзывчивость UI** — пользователь не ждёт выполнения всех фоновых задач
- **Устойчивость к сбоям** — отказ версионирования не приводит к потере заметки
- **Масштабируемость** — каждый сервис можно масштабировать независимо
