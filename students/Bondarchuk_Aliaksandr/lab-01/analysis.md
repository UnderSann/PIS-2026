# Анализ границ ответственности и транзакционных операций

## 1. Загрузка файла и сохранение метаданных заметки в БД

**Тип операции:** Синхронная

**Поведение при ошибке:**
- Полный откат (ROLLBACK) транзакции в базе данных
- Если сохранение метаданных не удалось — файл не привязывается к заметке

**Retry-стратегия:**
- Не применяется (контролируется механизмами самой БД)

**Идемпотентность:**
- Да. Проверка по уникальному ID запроса или хешу содержимого файла, чтобы избежать дубликатов при повторной отправке

**Обоснование:**
Это ядро всей операции. Если заметка не сохранилась в системе — все дальнейшие действия теряют смысл. Атомарность на этом этапе критически важна для целостности данных.


## 2. Сохранение файла в хранилище

**Тип операции:** Синхронная

**Поведение при ошибке:**
- Да, система удаляет частично загруженный файл
- Освобождает занятое место в хранилище

**Retry-стратегия:**
- 3 повторные попытки
- Экспоненциальная задержка: 1с → 2с → 4с

**Идемпотентность:**
- Да. Используется комбинация хеша файла и оригинального имени — если файл уже существует с таким же содержимым, повторная загрузка не создаст дубликат

**Обоснование:**
Файл — это основное содержимое заметки. Без него заметка бесполезна. Retry необходимы при временных сбоях диска или сети.


## 3. Создание записи в сервисе версионирования

**Тип операции:** Асинхронная

**Поведение при ошибке:**
- Немедленного отката нет
- Задание помещается в очередь для повторной обработки
- Основная заметка уже сохранена и доступна пользователю

**Retry-стратегия:**
- 5 повторных попыток с экспоненциальной задержкой
- При исчерпании попыток — в Dead Letter Queue (DLQ) для ручного анализа

**Идемпотентность:**
- Да. По комбинации ID заметки и timestamp версии — повторный запрос не создаст дубликат версии

**Обоснование:**
Версионирование — важная, но не критичная функция. Пользователь может начать работать с заметкой сразу, а история версий создастся позже. Это повышает отзывчивость системы.


## 4. Рассылка команд синхронизации на устройства

**Тип операции:** Асинхронная

**Поведение при ошибке:**
- Нет (best-effort доставка)
- Используется брокер сообщений с гарантиями доставки

**Retry-стратегия:**
- Встроена в брокер сообщений (RabbitMQ/Kafka)
- Автоматические повторные попытки при сбоях

**Идемпотентность:**
- Да. Каждое событие синхронизации имеет уникальный ID — устройства игнорируют уже обработанные команды

**Обоснование:**
Устройства могут быть офлайн. Они получат обновления при подключении. Немедленная доставка не требуется, что позволяет сделать этот шаг полностью асинхронным.


## Общий вывод по транзакционным границам

Основная транзакция заканчивается после успешного сохранения файла и метаданных заметки. Все остальные операции (версионирование, синхронизация) вынесены за её пределы и обрабатываются асинхронно. Такой подход даёт три ключевых преимущества:

- **Высокая отзывчивость UI** — пользователь не ждёт выполнения фоновых задач
- **Устойчивость к сбоям** — отказ версионирования не приводит к потере заметки
- **Масштабируемость** — каждый сервис можно развивать и масштабировать независимо
