### 2. Use-case описание

👉 **Ссылка на файл:** [use-case.md](use-case.md)

**Основной сценарий:** Создание новой заметки из локального .txt файла

**Первичный актор:** Зарегистрированный пользователь системы Notes Sync

**Цель:** Пользователь хочет создать новую заметку, загрузив существующий текстовый файл со своего устройства, чтобы затем иметь к ней доступ с любого другого устройства

**Краткое описание основного потока:**
1. Пользователь выбирает .txt файл на своём устройстве
2. Система загружает файл на сервер
3. Система сохраняет метаданные заметки в БД
4. Система создаёт первую версию заметки
5. Система инициирует синхронизацию на другие устройства
6. Пользователь видит сообщение об успехе

**Альтернативные потоки:**
- Пользователь выбрал файл неверного формата (не .txt) → система возвращает ошибку

**Исключительные ситуации:**
- Ошибка сохранения файла в хранилище
- Сервис версионирования временно недоступен (компенсация через очередь)

### 5. Анализ границ ответственности

👉 **Ссылка на файл:** [analysis.md](analysis.md)

#### 5.1. Транзакционные границы

| Операция | Синхронная/Асинхронная | Откат при ошибке | Retry-стратегия | Идемпотентность |
|----------|------------------------|------------------|-----------------|-----------------|
| **Загрузка файла и сохранение метаданных заметки в БД** | Синхронная | Да (ROLLBACK транзакции БД) | Нет (контролируется БД) | Да (проверка по уникальному ID запроса или хешу файла) |
| **Сохранение файла в хранилище** | Синхронная | Да (удаление загруженного файла) | 3 попытки с эксп. задержкой (1с, 2с, 4с) | Да (хеш файла + имя) |
| **Создание записи в сервисе версионирования** | Асинхронная | Нет (помещаем в очередь для повторной попытки) | 5 попыток с эксп. задержкой, затем в очередь мертвых писем (DLQ) | Да (ID заметки + timestamp версии) |
| **Рассылка команд синхронизации на устройства** | Асинхронная | Нет (best-effort через WebSocket/очередь) | Встроено в брокер сообщений (RabbitMQ/Kafka) | Да (уникальный ID события синхронизации) |

#### 5.2. Обработка исключительных ситуаций

**Реализовано стратегий обработки:** 3

**Примеры:**

##### Исключительная ситуация 1: **Таймаут сервиса версионирования**

- **Условие возникновения:** Version Service не отвечает в течение 3 секунд при попытке создать первую версию заметки
- **Обнаружение:** Notes API получает TimeoutException при вызове POST /api/versions
- **Реакция:** Система НЕ откатывает создание заметки (файл сохранён, запись в БД создана), но помещает задание на создание версии в очередь для повторной обработки. Заметка в БД помечается флагом versioning_pending = true
- **Компенсация:** Фоновый процесс (Worker) периодически забирает задания из очереди и повторяет попытки создать версию (до 5 раз с экспоненциальной задержкой). При исчерпании попыток задание перемещается в Dead Letter Queue для ручного анализа администратором
- **Уведомление пользователя:** "Заметка сохранена, но история версий временно недоступна. Мы попробуем создать её автоматически в течение нескольких минут."

##### Исключительная ситуация 2: **Конфликт версий при синхронизации (Race Condition)**

- **Условие возникновения:** Два устройства пользователя (например, телефон и ноутбук) одновременно отредактировали одну и ту же заметку в офлайн-режиме и при выходе в онлайн пытаются сохранить изменения, отправляя одну и ту же базовую версию
- **Обнаружение:** При выполнении UPDATE в БД с условием WHERE id = ? AND version = :oldVersion обнаруживается, что затронуто 0 строк (optimistic lock сработал)
- **Реакция:** Сервер возвращает клиенту ошибку 409 Conflict и в теле ответа присылает актуальную версию заметки (с изменениями с другого устройства)
- **Компенсация:** Обе версии заметки сохраняются в системе как конфликтующие. Пользователю предлагается вручную выбрать нужные изменения или объединить их
- **Уведомление пользователя:** "Обнаружен конфликт в заметке 'Название'. На другом устройстве были внесены изменения. Пожалуйста, выберите версию для сохранения или объедините изменения вручную."

##### Исключительная ситуация 3: **Таймаут сервиса синхронизации**

- **Условие возникновения:** Sync Service не отвечает при попытке отправить событие о создании новой заметки для рассылки на устройства пользователя
- **Обнаружение:** Notes API ловит исключение (TimeoutException или ConnectException) при вызове Sync Service
- **Реакция:** Событие не теряется — оно помещается в очередь сообщений (брокер) для повторной отправки. Основная транзакция создания заметки считается успешной
- **Компенсация:** Брокер сообщений (например, RabbitMQ или Kafka) автоматически выполняет повторные попытки доставки события. При временной недоступности сервиса сообщение остаётся в очереди до восстановления соединения
- **Уведомление пользователя:** "Заметка сохранена. Синхронизация с другими устройствами будет выполнена автоматически при восстановлении соединения."

## Контрольные вопросы

**Подготовка к защите:**

1. **Что такое транзакционная граница? Где она проходит в вашем сценарии?**
   
   Транзакционная граница — это логическая единица работы, которая должна быть выполнена полностью или не выполнена вовсе (свойство атомарности ACID). Внутри транзакционной границы все операции либо успешно завершаются, либо полностью откатываются при ошибке.
   
   В моём сценарии основная транзакционная граница проходит между шагами 4-6 (отправка файла на сервер → сохранение в File Storage → создание записи в БД). Эти операции выполняются атомарно: если сохранение файла или запись в БД не удались, происходит полный откат — файл удаляется из хранилища, запись в БД не создаётся. Вторая транзакционная граница (уже асинхронная) включает вызов Version Service и Sync Service, которые могут выполняться независимо.

2. **Почему операция "сохранение файла" выбрана синхронной, а "версионирование" - асинхронной?**
   
   **Сохранение файла** выбрано синхронным, потому что это критическая операция — без файла заметка не имеет смысла для пользователя. Пользователь должен получить подтверждение, что его данные сохранены, прежде чем закрыть вкладку или приложение. Кроме того, файл является источником истины (source of truth) для всей заметки.
   
   **Версионирование** выбрано асинхронным, потому что:
   - Пользователь может начать работать с заметкой сразу, а история версий создастся позже
   - Это повышает отзывчивость UI — пользователь не ждёт выполнения фоновой операции
   - При временной недоступности Version Service мы не теряем данные пользователя, а просто откладываем создание версии
   - Асинхронная очередь позволяет переживать пиковые нагрузки и временные сбои

3. **Как обеспечить идемпотентность при повторных запросах?**
   
   Идемпотентность обеспечивается несколькими способами:
   
   **1. Использование idempotency key (уникального ключа идемпотентности):**
   - Клиент генерирует уникальный UUID для каждого запроса на создание заметки
   - Сервер проверяет, выполнялся ли уже запрос с таким ключом
   - Если выполнялся — возвращает сохранённый результат, не создавая дубликат
   
   **2. Хеширование содержимого:**
   - При сохранении файла вычисляется его хеш (SHA-256)
   - Если файл с таким хешем уже существует у пользователя — возвращаем существующую заметку
   
   **3. Комбинация идентификаторов:**
   - Для версионирования используем связку note_id + timestamp версии
   - Повторный запрос на создание той же версии будет проигнорирован
   
   **4. Проверка состояния:**
   - Перед выполнением операции проверяем, не находится ли ресурс уже в целевом состоянии
   - Например, перед созданием версии проверяем, не создана ли она уже

4. **Что произойдёт, если внешний сервис вернёт ошибку после частичного выполнения операции?**
   
   В зависимости от критичности сервиса применяются разные стратегии:
   
   **Для критических сервисов (File Storage):**
   - Происходит полный откат транзакции — созданная запись в БД удаляется, загруженный файл стирается
   - Пользователю возвращается ошибка с предложением повторить попытку
   
   **Для некритичных сервисов (Version Service, Sync Service):**
   - Основная транзакция НЕ откатывается — заметка считается созданной
   - Операция помещается в очередь для повторной попытки (retry queue)
   - В БД проставляется флаг (например, versioning_pending = true)
   - Пользователь получает предупреждение, что операция будет выполнена позже
   - Фоновый процесс периодически пытается выполнить отложенные операции
   
   **Для сервисов с гарантированной доставкой:**
   - Используется брокер сообщений (RabbitMQ/Kafka) с подтверждениями
   - Сообщение остаётся в очереди до успешной обработки

5. **Как система обнаружит, что внешний сервис недоступен?**
   
   Система использует несколько механизмов обнаружения недоступности:
   
   **1. Таймауты (Timeouts):**
   - На каждый HTTP-вызов устанавливается таймаут (например, 3 секунды для Version Service)
   - Если ответ не получен за это время — выбрасывается TimeoutException
   
   **2. Circuit Breaker (Предохранитель):**
   - При накоплении определённого количества ошибок (например, 5 за 10 секунд) схема размыкается
   - Все последующие вызовы мгновенно завершаются ошибкой без попытки соединения
   - Через некоторое время (таймаут восстановления) делается пробный запрос для проверки
   
   **3. Health Checks:**
   - Система периодически опрашивает эндпоинты /health внешних сервисов
   - При недоступности сервис помечается как нездоровый, трафик временно не направляется
   
   **4. Retry с экспоненциальной задержкой:**
   - При первой ошибке делается повторная попытка через 1 секунду
   - При повторных ошибках интервал увеличивается (2с, 4с, 8с...)
   - Если все попытки исчерпаны — сервис считается недоступным
   
   **5. Логирование и мониторинг:**
   - Все ошибки соединения логируются с указанием времени и типа ошибки
   - Система мониторинга (например, Prometheus + Grafana) отслеживает процент ошибок и оповещает администраторов

6. **Какие данные нужно логировать для диагностики сбоев?**
   
   Для эффективной диагностики сбоев необходимо логировать следующие данные:
   
   **Обязательные поля:**
   - **Timestamp** — точное время события (включая часовой пояс или UTC)
   - **Request ID / Correlation ID** — уникальный идентификатор запроса для связывания всех связанных логов
   - **User ID** — идентификатор пользователя (если доступен)
   - **Operation type** — тип операции (CREATE_NOTE, UPLOAD_FILE, CREATE_VERSION и т.д.)
   - **Resource ID** — идентификатор затронутого ресурса (note_id, file_id, version_id)
   
   **Техническая информация:**
   - **Service name** — название сервиса, сгенерировавшего лог
   - **Endpoint / Method** — какой эндпоинт вызывался (POST /api/notes)
   - **HTTP status code** — код ответа (200, 409, 500, 504)
   - **Error message / Stack trace** — сообщение об ошибке и стек вызовов
   - **Duration** — время выполнения операции
   
   **Контекст ошибки:**
   - **Exception type** — тип исключения (TimeoutException, SQLException)
   - **Retry count** — номер попытки при повторных вызовах
   - **Fallback triggered** — был ли применён запасной механизм
   - **Compensation action** — какое компенсирующее действие выполнено
   
   **Сетевые параметры:**
   - **Client IP** — IP-адрес клиента
   - **User Agent** — информация о браузере/устройстве
   - **Request payload** — тело запроса (без чувствительных данных, либо замаскированное)
