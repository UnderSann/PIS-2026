# Architecture Overview - Note Service

## 1. Гексагональная архитектура (Ports & Adapters)

Note Service построен по принципам **гексагональной архитектуры** (также известной как Ports & Adapters). Этот подход обеспечивает:

- **Независимость от внешних систем** — бизнес-логика не зависит от БД, API или файлового хранилища
- **Тестируемость** — возможность тестировать изоляцию с моками
- **Гибкость** — замена адаптеров без изменения ядра

### Входящие порты (Inbound Ports) — API для клиентов

| Порт | Назначение | Входные данные | Выходные данные |
|------|------------|----------------|-----------------|
| `ICreateNoteUseCase` | Создание новой заметки | `CreateNoteCommand` | `Note` |
| `IGetNoteUseCase` | Получение заметки по ID | `GetNoteQuery` | `Note` |
| `IUpdateNoteUseCase` | Обновление существующей заметки | `UpdateNoteCommand` | `Note` |
| `IDeleteNoteUseCase` | Удаление заметки | `DeleteNoteCommand` | `void` |

### Исходящие порты (Outbound Ports) — зависимости от внешнего мира

| Порт | Назначение | Методы | Ответственность |
|------|------------|--------|-----------------|
| `INoteRepository` | Хранение метаданных заметок | `save()`, `findById()`, `findByOwner()`, `delete()` | CRUD операции с заметками в БД |
| `IFileStorage` | Хранение файлов с контентом | `upload()`, `download()`, `delete()` | Сохранение и загрузка файлов |
